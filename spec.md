# 新潟県温泉アプリ「湯めぐり新潟」アプリ制作プロンプト

---

## ■ プロジェクト概要

新潟県内の温泉地に特化したモバイルアプリ（iOS / Android）を制作してください。
コアコンセプトは「混雑状況をリアルタイムで把握し、快適に温泉巡りができる」こと。
ターゲットは新潟県内外の温泉愛好家・観光客。

---

## ■ デザイン方針

- カラーテーマ：湯けむりを連想させる落ち着いたトーン（ダークネイビー × 温かみのある琥珀色 × オフホワイト）
- フォント：日本語は視認性の高いゴシック体、英数字はモダンなサンセリフ
- ダークモード対応必須（設定でオン/オフ切替）
- アイコン・UIは温泉・和をモチーフにしたオリジナルデザイン
- 混雑状況の色分け表記：
  - 空き → 緑文字（例：`#27AE60`）
  - 普通 → 青文字（例：`#2980B9`）
  - 混んでいる → オレンジ文字（例：`#E67E22`）

---

## ■ 技術要件

- プラットフォーム：iOS / Android（**React Native + Expo** で実装。Firebaseを一切介さずSupabase単独で完結させるため、ジオフェンシング（`expo-location`）・画像圧縮（`expo-image-manipulator`）・プッシュ通知（Expo Push Notification Service）を全てExpoエコシステムで統一する）
- GPS機能：位置情報の取得と自動チェックイン処理
- 自動チェックイン：温泉地の中心座標から**半径100m以内**に入ると自動チェックイン
- 混雑情報更新間隔：**5分ごと**にリアルタイム更新
- **認証：Supabase Auth（Google OAuthプロバイダ）を使用**
  - Firebase Authenticationは使用しない。DB・Storage・Authを Supabase に一本化する
  - モバイルネイティブGoogleサインイン → IDトークンをSupabaseに渡し `signInWithIdToken` でセッション確立する方式
    - React Native (Expo)：`@react-native-google-signin/google-signin` を使用
    - Flutter：`google_sign_in` パッケージを使用
  - ユーザーテーブルは `profiles.id = auth.users.id`（uuid）の1:1構成とし、RLSポリシーは `auth.uid()` を主体に設計する
- 画像ストレージ：**Supabase Storage（無料枠）** を使用
  - 無料枠：1GB ストレージ（容量節約のため画像は必ず圧縮してからアップロード）
  - アップロード前に画像を**最大1280px・品質80%に自動圧縮**する処理を必ず実装
  - **圧縮処理はクライアント側（アプリ内）で実行する**：モバイルアプリは`expo-image-manipulator`でリサイズ&圧縮してからアップロード。理由：Supabase Edge Functions（軽量ランタイム）はSharp等の重い画像処理に不向きで実行時間・メモリ制約のリスクがあるため、アップロード元での圧縮が確実
  - 管理者ダッシュボード（Web）側もアップロード前にブラウザ内で圧縮（`browser-image-compression`等のライブラリでcanvasベース圧縮）を実行してから送信する
  - 管理者ダッシュボードからアップロード → Supabase StorageにURLが発行 → DBに保存 → アプリに表示
  - データベースもSupabaseに統一（Auth含め完全にSupabaseへ統一）
- ユーザーアイコン画像：端末内フォトライブラリから選択 → `expo-image-manipulator`で圧縮処理後にSupabase Storageにアップロード

## ■ GPS・ジオフェンシング実装方式（重要・アーキテクチャ決定事項）

継続的なバックグラウンド位置情報ポーリングではなく、**OSネイティブのGeofencing（Region Monitoring）APIにジオフェンス登録を任せる**方式で実装する。

### 制約と対応方針

| 制約 | 内容 | 対応 |
|---|---|---|
| iOSのRegion Monitoring上限 | 1アプリあたり**最大20リージョン**まで同時監視可能 | 全温泉地を常時監視せず、**現在地から近い温泉地20件以内**のみを動的にジオフェンス登録する。ユーザーが大きく移動したタイミング（Significant Location Change等）で登録リストを再計算する |
| Androidのジオフェンス上限 | 1アプリあたり最大100件 | iOSほど厳しくないが、同様に近隣施設優先で動的登録する設計に統一する |
| Region Exitイベントの取りこぼし | アプリがOSにkillされると圏外退出イベントを検知できない場合がある | **サーバー側TTLによる強制チェックアウトを実装する**（例：最終ハートビート/確認通信から30分経過したユーザーは自動チェックアウト扱いとし、混雑カウントから除外する）。ジオフェンスのExitイベントだけに依存しない |
| iOSのバックグラウンド位置情報権限 | `Always`権限（`NSLocationAlwaysAndWhenInUseUsageDescription`）が必須。App Store審査で用途説明が必要 | 「温泉施設の混雑状況をリアルタイムに反映するため」という利用目的を明記する |
| Androidのバックグラウンド位置情報権限 | Android 10+では`ACCESS_BACKGROUND_LOCATION`を`ACCESS_FINE_LOCATION`と別に追加リクエストする必要がある | 権限リクエストフローを2段階で設計する |
| 権限が「使用中のみ」または拒否の場合 | 自動チェックイン/アウトが機能しない | 詳細ページ内の「チェックインする」ボタンによる手動チェックインへ確実にフォールバックする（既存仕様どおり） |

### 使用ライブラリ
- `expo-location` の `Location.startGeofencingAsync()`（iOS/Android両対応）を採用。より高度な制御が将来必要になった場合は `react-native-background-geolocation`（有償）への切り替えを検討

- 各レビューカードに**通報ボタン**を設置（「…」メニュー内）

  1. スパム・宣伝

  3. 管理者が内容を確認し、「問題なし」→通常表示に復帰、「削除」→次項のプレースホルダーに切り替え
- 管理者が削除したレビューは「このレビューは削除されました」と表示（完全非表示ではなくプレースホルダー表示）

## ■ GPS・チェックイン／チェックアウト仕様

- アプリ起動時にGPS使用許可をリクエスト
- 許可後、**現在地から近い温泉地（iOS上限を踏まえ最大20件）を対象に動的にジオフェンス（Region Monitoring）を登録**する（詳細は「GPS・ジオフェンシング実装方式」章を参照）
- 各温泉地に緯度・経度の中心座標を登録
- **ユーザーが中心座標から半径100m以内に入ると自動チェックイン**
- **ユーザーが半径100m圏外に出ると自動チェックアウト**（混雑カウントから即時除外）
- **上記に加え、Exitイベント取りこぼし対策としてサーバー側TTLによる強制チェックアウトを実装する**（例：最終ハートビートから30分経過で自動チェックアウト）
- チェックイン時にプッシュ通知（通知オンの場合）：「○○温泉にチェックインしました！」
- チェックアウト時にプッシュ通知（通知オンの場合）：「○○温泉を退場しました。またのご利用をお待ちしています」
- 同一施設への重複チェックインは**2時間以内はブロック**（圏外→圏内の短時間出入りで誤カウントを防止）
- GPS権限がオフの場合は手動チェックイン画面へ誘導（詳細ページ内に「チェックインする」ボタンを表示）

## ■ プッシュ通知の実装基盤

- **Expo Push Notification Service** を使用（Expo構成のため、iOS/Android個別のFCM/APNs設定を直接扱わずに統一的に送信できる）
- トリガーは **Supabase Database Webhooks / Edge Functions**：対象テーブル（チェックイン/チェックアウトイベント、レビュー通報ステータス変更、施設の`is_temporarily_closed`フラグ変更）へのINSERT/UPDATEをトリガーに、Edge FunctionからExpo Push APIへ通知リクエストを送信
- 各ユーザーの`profiles`テーブルに`expo_push_token`を保持し、通知オフのユーザーには送信しない

## ■ 管理者・オーナー権限管理（RBAC）

- `profiles.role`：`user` / `owner` / `admin` の3種
- `owner_onsen_links`テーブル（`owner_id`, `onsen_id`の多対多）で、施設オーナーが担当する施設を紐付け（1オーナーが複数施設を持つ、または1施設に複数オーナーが紐づくケースに対応）
- 権限の適用範囲：
  - `admin`：全施設・全レビューに対しての編集・削除権限（RLSで`role = 'admin'`を許可条件に含める）
  - `owner`：**料金・営業時間・休業情報のみ編集可**という制約があるため、生テーブルへの直接UPDATEをRLSだけで列単位に絞るのではなく、**専用のSupabase Edge Function（RPC）経由で許可フィールドのみ更新を受け付ける**方式にする（Postgres RLSは行単位の制御が基本で列単位の制限が煩雑なため）
  - `user`：自分のレビュー・お気に入り・プロフィールのみ編集可（通常のRLS `auth.uid()`制御）

## ■ オフライン時のフォールバック

- **チェックイン/チェックアウト**：ジオフェンスイベント発生時にオフラインであれば、イベントをローカル（`expo-sqlite`または`AsyncStorage`）にキューイングし、再接続時に自動同期する
- **混雑状況表示**：直近取得済みのデータをローカルキャッシュ表示し、画面上に「オフライン - 最終更新 XX:XX」と明示する。5分毎の自動更新は再接続時に再開
- **お気に入り等のユーザー操作**：ローカルで即時反映（オプティミスティックUI）した上で再接続時に同期。競合時はサーバー側のデータを正として上書きする

## ■ データ構造（参考）

### 「温泉地」の粒度

**「温泉地」1件 = 個別の入浴施設単位**（例：「ホテル○○の日帰り温泉」「××旅館 大浴場」）と定義する。「月岡温泉」「越後湯沢温泉」のような温泉街・エリア名は、施設データ内の`area`フィールド（グルーピング用）として保持し、探す画面の地域フィルターや温泉街単位でのまとめ表示に使う。越後湯沢温泉のように多数の施設が存在するエリアは、施設ごとに個別エントリを作成する（初期シードでは主要施設を代表1〜数件ずつ登録し、順次拡充する）。

### 温泉地（施設）データ

```json
{
  "id": "tsukioka_hotel_a_001",
  "name": "ホテル泉慶 日帰り温泉",
  "area_name": "月岡温泉",
  "prefecture": "新潟県",
  "city": "新発田市",
  "area": "月岡",
  "region": "下越",
  "latitude": 37.8456,
  "longitude": 139.3812,
  "hours": "10:00〜21:00",
  "features": {
    "rotenburo": true,
    "sauna": true,
    "restaurant": true,
    "parking": true
  },
  "description": "硫黄成分を豊富に含む緑色の湯が特徴。美肌効果で知られる。",
  "components": "含硫黄-ナトリウム-塩化物泉",
  "effects": "美肌、神経痛、筋肉痛、冷え性",
  "price": {
    "adult": 800,
    "child": 400,
    "child_condition": "小学生（6〜12歳）。未就学児は無料"
  },
  "capacity": {
    "total": 25,
    "male": 12,
    "female": 12
  },
  "phone": "0254-32-XXXX",
  "website": "https://example-tsukioka.jp",
  "regular_holiday": "毎週火曜日",
  "winter_closure": { "start": "12-01", "end": "03-31", "enabled": false },
  "is_temporarily_closed": false,
  "photos": ["url1", "url2"]
}
```

- `area_name` / `area`：温泉街・エリア名（表示・グルーピング用。ホーム画面や探す画面での「エリアまとめ表示」に使用）
- `region`：上越／中越／下越（探す画面の地域フィルターに使用。市区町村から自動判定 or 手動登録）
- `price.child_condition`：対象年齢は施設ごとに異なるため自由記述で保持する
- `capacity`：混雑率算出の分母。男女別混雑率のため`male`/`female`を個別に持つ（`male + female`は`total`と厳密に一致しなくてよい。性別未設定ユーザー分の余裕を`total`側に含めるため）
- **管理者ダッシュボードでの新規登録時、既存施設の中心座標から300m以内に新規座標がある場合は警告表示し、座標・半径の見直しを促す**（同一エリア内で施設が密集し、ジオフェンスが重複・誤検出するのを防ぐため）

### 混雑度の色分けしきい値

`congestion_rate = round(users_count / capacity.total * 100)`（男女別も同様に各capacityで算出）

| 混雑率 | 表示 | 色 |
|---|---|---|
| 0〜39% | 空き | 緑 `#27AE60` |
| 40〜69% | 普通 | 青 `#2980B9` |
| 70%以上 | 混んでいる | オレンジ `#E67E22` |

100%を超える場合（定員超過）も実数のまま表示し、色はオレンジのまま据え置く（UI上は100%で頭打ちにせず実数を出す）。

### 混雑データ（5分毎更新）

## ■ 温泉地データの更新フロー

### 更新ルート（3種類）

| ルート | 担当者 | 内容 |
|--------|--------|------|
| 管理者による直接編集 | アプリ運営者 | 管理者ダッシュボード（Web）から全項目を編集可能 |
| 施設オーナー申請 | 温泉施設側 | オーナー登録後、自施設の料金・営業時間・休業情報のみ編集可 |
| ユーザーからの情報提供 | 一般ユーザー | 詳細画面の「情報を修正する」ボタンから変更提案を送信 → 管理者が承認後に反映 |

### 休業・臨時休業の反映
- 管理者またはオーナーが「休業フラグ」をオンにすると、詳細画面・カード一覧の両方に「現在休業中」バナーを即時表示
- 冬季休業は開始日・終了日を登録し、期間中は自動でバナー表示

### データ更新時の通知
- お気に入り登録済みの施設が「臨時休業」になった場合、プッシュ通知で該当ユーザーに通知（通知オンの場合）

## ■ 追加開発メモ

- 温泉データのシード：新潟県内の主要温泉地（月岡温泉、瀬波温泉、咲花温泉、越後湯沢温泉、赤倉温泉、松之山温泉、栃尾又温泉、岩室温泉など）を初期データとして登録
- 混雑データの収集方法：チェックイン／チェックアウト（100m圏外退出）をベースに在館ユーザー数をリアルタイム集計
- 管理者用ダッシュボード（Web）：温泉地の追加・編集、休業管理、通報されたレビューの確認・削除
- 画像管理：Supabase Storageを使用。管理者ダッシュボードから写真をアップロード（最大1280px・品質80%に自動圧縮） → URLをDBのphotos配列に保存 → アプリのカード・詳細画面に表示。開発初期はUnsplash APIのダミー画像で代替し、リリース前に正式写真へ差し替える。データベース・認証ともSupabaseに統一

## ■ 決定事項サマリー（レビューで確定した仕様）

| 論点 | 決定内容 |
|---|---|
| 実装フレームワーク | React Native + Expo（Firebase不使用、Supabaseに完全統一） |
| 認証 | Supabase Auth（Google OAuth、ネイティブID Token交換） |
| GPS/ジオフェンス | `expo-location`の動的Region Monitoring（iOS20件上限を考慮し近隣施設のみ登録）＋サーバー側TTLによる強制チェックアウト |
| 「温泉地」の粒度 | 個別入浴施設単位（温泉街名は`area_name`として保持） |
| 混雑率算出 | `capacity`（定員）フィールドを追加し`users_count / capacity`で算出。しきい値は0-39%緑/40-69%青/70%以上オレンジ |
| レビュー通報対応 | 3件到達 or 誹謗中傷カテゴリ1件で自動的に「審査中」プレースホルダー表示に切替 |
| プッシュ通知 | Expo Push Notification Service + Supabase Edge Functions/Webhooks |
| 管理者/オーナー権限 | `profiles.role`によるRBAC。オーナーの限定編集はEdge Function経由 |
| 画像圧縮 | クライアント側（`expo-image-manipulator`／管理画面は`browser-image-compression`）で実行 |
| オフライン対応 | ローカルキュー＋再接続時同期、キャッシュ表示 |
